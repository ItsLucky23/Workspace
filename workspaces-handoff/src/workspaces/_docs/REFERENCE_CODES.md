# Reference Codes — inlined G#/B#/DH binding definitions + B-xx → owning-doc coverage matrix

> **Why this file exists.** The build-docs (`01`–`07`, `SETUP`, `INDEX`, the 24 `features/*`) cite framework-gap codes (`G#`), decision codes (`B-##` / `B-O#`), and decision-handoff codes (`DH#`) that historically live in the now-**deprecated** `handoff/` folder (`FRAMEWORK_GAPS.md`, `BESLISSINGEN.md`, `DATAMODEL.md`). Per **Q-INF-REFCODES** (recommendation accepted 2026-06-04) the `handoff/` folder is frozen — *immutable, not safe to delete* — until its codes are inlined HERE. This file is that inlining: a **binding, self-contained** definition of every code the `_docs` actually cite, so the build-docs no longer depend on `handoff/`. From here on, `_docs` cite **[REFERENCE_CODES → G#/B#]** and read the body below; the `handoff/` originals are historical provenance only.
>
> **Scope.** Only codes *actually cited in `_docs`* are inlined (grep-verified 2026-06-04). The full `handoff/` catalog (G1–G29, B-01…B-39, B-O1…B-O8, PR-1…PR-4) is broader; uncited codes are intentionally omitted to keep this binding and lean. If a future doc needs an uncited code, inline it here first (same protocol).
>
> **Carries:** `Q-INF-REFCODES` (this file is its deliverable) and `Q-INF-GAPS-RECONCILE` (the G# Status column below is the reconciliation that `FRAMEWORK_GAPS.md` rewrite realizes — RESOLVED-FW codes already ship in `packages/`; a builder must NOT re-implement them). No new verbs. No new persistence introduced by this doc — it is a reference index only.
>
> **Status legend (G# only).**
> | Status | Meaning |
> |---|---|
> | **RESOLVED-FW** | Already shipped in `packages/`. A builder reading the stale `FRAMEWORK_GAPS.md` would write redundant code — **don't**. Proof-path = package + exported helper. |
> | **APP-BUILD** | A real gap the Workspaces app builds on top of framework primitives. Owned by a named `_docs` section. |
> | **EXTERNAL-INFRA** | Resolved by an out-of-band infra choice (Caddy / Atlas Local / Docker / bullmq-or-in-process), not framework code. |

## 3. Decision-handoff codes (DH) — binding definitions

> `DH5` is **not** a single decision — in `handoff/DATAMODEL.md` it is a banner used four times to formalize four distinct schema-shaping rules. The `_docs` cite "DH5" generically; each citation means *one* of the four sub-rules below. Inlined so the `_docs` no longer reach into `handoff/DATAMODEL.md`.

| Code (sub-rule) | Binding definition | Cited by |
|---|---|---|
| **DH5 — stage/status shape** | A ticket = `{ stageId, statusKey }`. **Stage** = pipeline step (column); **status** = state within that step (pill). Strictly two levels. | Drift: typed `StageKind` (not the fixed 7-literal union, not bare string) per `Q-DATA-STAGEID`; closed universal status enum (`idle\|needs-input\|busy\|done\|paused\|stuck`) per `Q-DATA-STATUS`. `02_PROTOCOL_AND_FLOW.md`, `03_BUILD_PHASE.md`. |
| **DH5 — commit-hash binding-timing** | `commitHash` is assigned at **worktree creation** (`git pull origin main` → that hash) and **frozen**: if `main` moves while the ticket is open, the ticket stays on its old hash (code + RAG context frozen). Only re-activation/new-worktree picks up the newest hash. Ticket/status context is live (§6.4). | `07_ORCHESTRATOR.md §worktree`, `23_PREVIEW_DEPLOYMENT.md` (preview reuses the frozen commitHash). |
| **DH5 — event-granularity rule** | One event per **meaningful milestone** (`B-21` coalescing): `command` = one per executed command; `file-change` = one per changed file (with +/− counts, not per chunk); `ai-message` = one per **completed** AI message (not per token — live tokens stream un-persisted); `mr`/`status-change`/`stage-move` = one per action. No sub-token rules. | `20_ACTIVITY_AND_EVENT_LOG.md`. |
| **DH5 — Workspace-AI autonomy-scope** | Per `B-23`: the Workspace-AI may **only produce proposals** (`WorkspaceSuggestion`) + write notes; never run scrum/git actions itself. Allowed proposal-types: link-tickets, suggest-epic, pipeline-config-finding, source-maintenance-warning. **Execution only after a user `accept`** (respecting RBAC — a Member cannot accept a config-changing proposal). Config-review heuristics live as app-rules, not hardcoded in the schema. | The same no-write-verb guarantee as `B-23`. `02_PROTOCOL_AND_FLOW.md`, `03_BUILD_PHASE.md`. |
