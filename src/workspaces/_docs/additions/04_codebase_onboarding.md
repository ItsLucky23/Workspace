> **Tier:** V1 (light) ¬∑ **Lane:** D + A ¬∑ **Status:** NEW (2026-06-11).

## 1. The gap this closes

The setup wizard ([`features/01`]) ends by handing a linked `Project` to the **build phase** ([`features/03`]), which clones the repo, lets the user mark each path **GENERATE** vs **LINK** ‚Üí `Project.linkedFiles[]`, snapshots RAG @ `commitHash`, and materializes the initial `InfoDoc` set the Assistant reads to author the pipeline. That flow assumes a **human (via the Assistant) drives the GENERATE/LINK split** ‚Äî `features/03` resolves it as *"the Assistant **proposes** the split; the user confirms; sensible fallback defaults apply if it doesn't"* (03.q1). For an existing, well-organized repo with a `CLAUDE.md` and a `README`, the defaults are fine.

For a **brand-new repo with nothing** ‚Äî no `CLAUDE.md`, no conventions doc, no project summary ‚Äî the build phase has nothing to LINK and no starter context to GENERATE from. The pipeline's quality layer ([`AI_QUALITY_AND_EVALS ¬ß2`]) leans on `customInstructions` (the per-turn `CLAUDE.md` domain rules) being *present and good*; on a fresh repo it is absent. The result is a cold-start: every stage-agent runs with an empty `CLAUDE.md`, produces lower-grounded output (P4 violations: invented files/APIs), and the human has to hand-author the starter docs before the pipeline is worth running.

**What's missing** is the *first-index* step the build phase never specifies: when a repo is connected for the first time, (a) **assess its health** so the user knows what they're working with, and (b) **auto-draft the starter context-doc set** (`CLAUDE.md` + a project summary + a conventions doc) so the pipeline starts warm ‚Äî without silently committing AI-authored docs into someone's repo. This addition is the explicit **first-index / onboarding** step that sits inside `features/03`'s GENERATE leg, between *clone* and *seed*.

## 2. Locked decision (generate + human-review gate)

**LOCKED:** First-index = **index the repo (RAG) + produce a repo-health report + auto-draft a starter `CLAUDE.md`/context-doc set, shown to the human for review/edit, committed to the repo ONLY on approval** (an `[control-API]` op ‚Üí the Conductor pushes). The three legs:

1. **Index (deterministic, orchestrator).** The build-phase clone @ `commitHash` runs the existing RAG snapshot ([`07 ¬ßD`], `features/03` step 4) exactly as today ‚Äî onboarding does not change the indexer.
2. **Health report (deterministic + LLM summary).** A deterministic scan (language/framework detection, test presence, repo size, lockfile, doc presence) produces a structured `RepoHealthReport`; the onboarding stage-agent adds a one-paragraph LLM summary. Deterministic where possible; the LLM only narrates what the scan already found (¬ß3.1).
3. **Draft (one-shot onboarding stage-agent).** A **new `onboarding` `AgentRole`** ([`03 ¬ß3`]) runs as a one-shot interactive PTY worker that reads the RAG context + the health report, uses [`REPO_CLAUDE.template.md`] as the scaffold, and emits the drafted doc set as its carry-over. It is a **reasoning-with-worktree** run: it reads the cloned worktree but its only *write* is the proposed doc bodies in the carry-over ‚Äî it does **not** commit (B-23: the agent never writes authoritative state).

**Review + approve.** The drafted docs are shown in the **same changes/editor surface used for ticket diffs** ([`features/07`] changed-files mode / [`CODE_EDITOR`]). The human reviews/edits, then **approves** ‚Äî a `[control-API]` op (`onboarding-approve`) that enqueues a Conductor action which commits the (possibly human-edited) docs into the repo under `Project.generatedDocsPath` and `CLAUDE.md` at the repo root, then pushes (same write-capable clone `features/03` already requires for GENERATE, 03.q3).

**Gate before any pipeline run.** **Nothing in the pipeline runs against the repo until onboarding is approved.** New tickets park at `idle` (the `aiEnabled=false` first-stage behavior `features/01` already relies on ‚Äî the board is usable while indexing); promoting a ticket into an `aiEnabled` stage is blocked while the workspace's onboarding is `pending`/`drafted` and not yet `approved`.

### DEFAULTs (flag if wrong)

- **DEFAULT ‚Äî the draft is a one-shot `onboarding`-role stage-agent run** (interactive PTY, [`07 ¬ßA`] launch path), **not** a deterministic template fill and **not** the user's chat Assistant. Rationale: the draft needs real codebase grounding (RAG + worktree reads) and the ¬ß2-style quality discipline a role prompt encodes; a pure template fill produces a generic `CLAUDE.md` that helps nobody. *Flag if you'd rather the Assistant draft it in-chat (cheaper, but no worktree/RAG grounding and no role-prompt quality contract).*
- **DEFAULT ‚Äî the health report is deterministic-first.** lang/framework/test/size/lockfile/doc-presence are scanned in the orchestrator (no LLM turn); the LLM only writes the human-readable summary. *Flag if you want the whole report LLM-authored (more narrative, but non-deterministic and burns a turn for facts a scan already has).*
- **DEFAULT ‚Äî review happens in the existing changes/editor surface**, not a bespoke onboarding screen. The drafted docs are "changed files" against the empty/baseline tree; the [`features/07`] stepper + the [`CODE_EDITOR`] read-only-then-editable flow apply unchanged. *Flag if you want a dedicated onboarding wizard step instead of reusing the diff surface.*

## 3. Build-ready mechanics

### 3.1 Health assessment (deterministic + LLM summary)

The health scan runs in the **orchestrator** (deterministic Conductor-driven work, no agent) right after the build-phase clone + RAG snapshot ([`features/03`] step 4, [`07 ¬ßD`]). It is a registered `OrchestratorCommandRegistry` command (`ai:onboarding-scan`, allow-listed like `ai:refresh-docs`, [`03 ¬ß2`]) ‚Äî **never raw shell**. It produces a structured `RepoHealthReport` from the worktree + the snapshot:

| Signal | Source (deterministic) | Why it matters |
|---|---|---|
| **languages** | file-extension histogram over the `.gitignore`-aware tree (`features/03` 03.q4 skips `node_modules`/`dist`/`.git`) | sizes the LSP/extension set the editor needs ([`CODE_EDITOR ¬ß6`]) |
| **framework / stack** | manifest detection (`package.json` deps, `go.mod`, `*.csproj`, `requirements.txt`, `Cargo.toml`) | drives the GENERATE'd summary + which conventions to scaffold |
| **lockfile present** | `package-lock.json`/`pnpm-lock.yaml`/`yarn.lock`/`go.sum`/‚Ä¶ | reproducible-build signal; missing lockfile is a health flag |
| **test presence** | glob for `*.test.*` / `*_test.go` / `tests/` / a `test` script in `package.json` | a no-tests repo flags the Test stage will start from zero |
| **repo size** | file count + byte total from the lazy-loaded tree (`features/03` large-repo guard, 03.q4) | sizes the RAG snapshot ETA + the CapacityManager budget |
| **existing docs** | presence of `README*`, `CLAUDE.md`, `CONTRIBUTING*`, `docs/` | LINK candidates; their absence is *why onboarding GENERATEs* |
| **commitHash / default branch** | `git rev-parse HEAD` (the frozen baseline, DH5) | binds the report + draft + RAG to one commit |

The **LLM summary** is one field on the report, produced by the onboarding stage-agent (¬ß3.2) ‚Äî *"a TypeScript + LuckyStack repo, 4.2k files, has a README but no `CLAUDE.md`, no tests yet; recommend GENERATE a project summary + conventions doc."* The deterministic block is the source of truth; the summary narrates it (DEFAULT, ¬ß2). The report renders into the build-phase **progress strip** (`features/03` user-flow) as a new **Health** panel above the GENERATE/LINK picker, and persists as a `RepoHealthReport` row (¬ß3.4) so a re-entered/mobile session re-syncs it ([`features/01`] failure-&-resume, B-37).

### 3.2 Draft generation (onboarding role; REPO_CLAUDE.template scaffold; PTY)

A **new built-in `AgentRole`** registered exactly like every other role ([`03 ¬ß3.2/¬ß3.3`] `registerAgentRole(...)`, zero core change):

```ts
registerAgentRole({
  key: 'onboarding',
  label: 'Codebase onboarding',
  description: 'First-index a new repo: assess health + draft the starter CLAUDE.md / context docs.',
  needsWorkspace: true,          // reads the cloned worktree + RAG; one-shot, no DEV processes
  defaultSkillKeys: ['rag', 'symbol-index'],   // ground the draft in real code (P4) ‚Äî same skills the build phase indexed
  defaultSourceIds: [],          // a fresh repo has no LINK docs yet ‚Äî that's the point
  defaultCommands: [/* read-only: ls/cat/grep + the allow-listed scan; NO push, NO write outside the draft */],
  defaultModelCfg: { /* a single capable model; this is a one-shot, not a multi-turn loop */ },
  outputSchema: /* base carry-over + draftedDocs:[{ path, role:'generate', body }] + healthSummary:string */,
  artifactKind: 'onboarding-draft',
  ingest: /* turns draftedDocs[] into the InfoDoc(source:'generated', status:'drafted') candidate set */,
  viewerComponent: undefined,    // falls back to the changes/editor surface (¬ß3.3) ‚Äî no bespoke viewer
  acceptFlow: 'diff-review',     // reuse the features/07 changed-files review gate
});
```

**Launch.** It runs through the standard one-shot worker launch ([`07 ¬ßA`] steps 1‚Äì7) under the orchestrator lease: pull ‚Üí freeze `commitHash` ‚Üí reuse-or-index the RAG snapshot (already done by the build phase for this commit; it *reuses*, never re-indexes, B-25) ‚Üí worktree ‚Üí container ‚Üí **interactive `claude` PTY** (never `claude -p`/Agent-SDK, [`01 ¬ß1`]). PTY-billing (hooks + the structured channel) applies unchanged ‚Äî it is an interactive node-pty session like any worker, so the P0.5 spike's billing/hook guarantees cover it; **no headless path**.

**Scaffold.** The role's `systemPromptTemplate` (an `AI_QUALITY_AND_EVALS ¬ß2`-style layer-1 body, owned by Lane D) instructs the agent to fill **[`REPO_CLAUDE.template.md`]** ‚Äî the template is the literal scaffold for the drafted root `CLAUDE.md`: it carries the carried-over LuckyStack working rules + the Workspaces invariants block + the `<!-- User project rules (none yet) -->` slot the agent populates from the detected stack/conventions. The agent additionally drafts a **project-summary** doc and a **conventions** doc (the `summary/conventions/glossary` slots `features/03` 03.q1 defaults to GENERATE). The prompt is bound by the ¬ß2 principles ‚Äî **P4 ground every claim in the RAG/worktree context** (never invent a convention the repo doesn't show), **P5 scope discipline** (draft only the starter set, don't editorialize), **P3 ask-don't-guess** (if the stack is ambiguous, `request_input` a `kind:'choice'` rather than guess a framework).

**Exit.** The agent finishes with `emit_carryover` carrying `draftedDocs[]` + `healthSummary` (the [`07 ¬ßA`] step-7 ‚Üí carry-over path). It **writes nothing to the repo** ‚Äî the carry-over *is* the proposal. If it cannot determine the stack, it `request_input`s and the onboarding parks at `needs-input` (the `features/01`/`02` question loop), surfaced on the board.

### 3.3 Review + approve‚Üícommit (reuse the changes/editor surface + CONTROL_API)

The drafted docs are reviewed in the **exact surface ticket diffs use** ‚Äî no new review UI:

- **Render as changed-files.** The `draftedDocs[]` (each `{ path, body }` against the baseline = absent file) populate the changed-files set the [`features/07`] changed-files mode renders. In the **interim** (UI-Builder/openvscode-server not yet mounted, [`features/07 ¬ßinterim`] / [`CODE_EDITOR ¬ß2.2`]) the read-only **`FileDiffViewer`/`DiffView`** shows each drafted doc as an all-additions diff at the frozen `commitHash`; the prev/next **stepper** walks the drafted set. Once [`CODE_EDITOR`]'s in-container openvscode-server is live, the same docs open in the real editor on the worktree (the draft is written to the worktree as uncommitted files so VS Code's native git decorations light them up, [`CODE_EDITOR ¬ß3`]).
- **Edit before approve.** The human edits the drafts directly (the [`CODE_EDITOR ¬ß4`] editable window ‚Äî onboarding is a `userMayEdit`-style draft-review state, not a running agent stage, so the worktree is the human's to edit). Edits land on the worktree; they ride the approve commit (¬ß5 of `CODE_EDITOR`, the "user edits ride the push-on-approval" mechanism ‚Äî here the push *is* the onboarding commit).
- **Approve = a `[control-API]` op.** Approval is the **`onboarding-approve`** op (a new row in the [`CONTROL_API ¬ß8`] catalogue): `login:true` + `preApiExecute` RBAC (the pipeline/config-edit capability, D30 ‚Äî a Member cannot approve onboarding, same tier as `skill-toggle`/`edit-pipeline`) ‚Üí **enqueue a Conductor action** ‚Üí `ControlAck{ accepted, signalSeq }`. The handler **never commits directly** ([`CONTROL_API ¬ß7`] enqueue-not-write); the Conductor drains the signal and:
  1. commits the (possibly human-edited) `draftedDocs[]` ‚Äî `CLAUDE.md` at repo root + the rest under `Project.generatedDocsPath` (default `docs/luckystack/`, `features/03` 03.q3) ‚Äî onto the default branch via the **write-capable** build-phase clone (`features/03` requires write/commit scope for GENERATE).
  2. `git push` to the GitLab remote (`forge.repoHosting.push()`, GitLab adapter only ‚Äî [`CODE_EDITOR ¬ß5`] push mechanics).
  3. materializes the approved docs as `InfoDoc{ source:'generated', status:'done' }` rows (the existing build-phase seed shape, `features/03` data) and flips the workspace's onboarding state ‚Üí `approved`, unparking the pipeline.
- **Reject / re-draft.** A reject is a free-text note that re-runs the onboarding role with the note as the `--resume` prompt (the [`features/07`] 07.q3 reject-reopens machinery ‚Äî "the note is the redirection"), `drafted ‚Üí busy`. Discarding instead is an `onboarding-skip` op (the user opts to hand-author; onboarding ‚Üí `approved` with an empty generated set, pipeline unparks).

**Optimistic UI.** Approve shows a transient "committing‚Ä¶" affordance; the authoritative state arrives over `ws-ai:*` at the confirming `seq` (the [`CONTROL_API ¬ß6.3`] merge-on-`seq` rule) ‚Äî the optimistic hint never wins over the Conductor's write.

**Feedback capture.** A human edit of a drafted doc before approve is a `PromptFeedback{ kind:'edit', roleKey:'onboarding' }` row ([`AI_QUALITY_AND_EVALS ¬ß5.1`]) ‚Äî the before‚Üíafter delta is a labeled correction that can be curated (Admin-promoted) into the onboarding role's few-shot bank, so drafts improve for *this team's* repos over time. No new mechanism: it rides the existing reject/edit‚Üífew-shot loop.

### 3.4 Data model (cite 04/04b)

Onboarding adds **one report row** and **two enum-value extensions** ‚Äî no new shape on the GENERATE leg (the draft reuses `Project.linkedFiles[]` + `InfoDoc`):

| Field / model | On | Type | Notes |
|---|---|---|---|
| `Project.generatedDocsPath` | `Project` | `String @default("docs/luckystack/")` | **reused** ‚Äî already in [`04b ¬ß13`] (D21) / `features/03` 03.q3; the approve-commit targets it (+ root `CLAUDE.md`). |
| `Project.linkedFiles[]` | `Project` | `{ path; role:'generate'\|'link' }[]` | **reused** ‚Äî [`04b ¬ß13`] (D9/D19); the approved drafts are `role:'generate'` entries. |
| `Project.onboardingState` | `Project` | `'pending' \| 'scanning' \| 'drafted' \| 'approved' \| 'skipped'` (NEW, default `'pending'`) | the gate (¬ß2): `aiEnabled`-stage promotion is blocked unless `approved`/`skipped`. Tenant-scoped on `Project` (= the workspace's one project, `features/01`). |
| `RepoHealthReport` | NEW model | `{ id, workspaceId, projectId, commitHash, languages Json, framework String?, hasTests Boolean, hasLockfile Boolean, fileCount Int, byteTotal Int, existingDocs String[], summary String?, createdAt }` | one row per first-index (re-index on a later onboarding re-run appends a new row keyed by `commitHash`). `summary` is the only LLM-authored field. `workspaceId` ‚Üí `runInTenant`-scoped like every tenant model ([`04 ¬ßintro`], [`04b ¬ß11c`]). |
| `InfoDoc.status` | `InfoDoc` | add `'drafted'` to the existing status set | a drafted-but-unapproved generated doc is `status:'drafted'`; approve flips it to `done` (the existing `indexing`/`done` generated-doc states `features/03` uses). Prototype: extend the `InfoDoc` interface. |
| `AgentRole` | registry (code, not DB) | `key:'onboarding'` | roles are **registered in code**, never stored per-workspace ([`04 ¬ß3`] note, [`03 ¬ß6`]) ‚Äî no migration. |

**INDEX delta:** `Project.onboardingState`; `RepoHealthReport` (new model); `InfoDoc.status += 'drafted'`. *(Fold into the `04`/`04b` field-sweep per the `features/01`/`03` "don't edit `04` directly" convention.)*

**CONTROL_API delta (cite `CONTROL_API ¬ß8`):** two new rows in the op catalogue ‚Äî `onboarding-approve` (`{ projectId }`, RBAC pipeline/config-edit D30, Conductor commits + pushes the drafts + flips state) and `onboarding-skip` (`{ projectId }`, same RBAC, Conductor flips state to `approved` with an empty generated set). Both are `login:true` + `preApiExecute` ‚Üí enqueue ‚Üí `ControlAck`. **None is a verb.**

## 4. Invariants honored

- **B-23 ‚Äî Conductor-only-writer.** The onboarding agent **proposes** the drafts via `emit_carryover`; the human **approves** via `onboarding-approve`; the **Conductor** commits + pushes. No LLM session ever writes the repo or flips `onboardingState`. The agent's worktree edits are not authoritative until the Conductor commits them on approve.
- **FROZEN verbs / writes via control-API.** Zero new structured-channel verbs ‚Äî the agent exits through existing `emit_carryover`/`request_input`; the approve/skip/re-draft writes are `[control-API]` ops ([`CONTROL_API ¬ß7/¬ß8`]), the only write path. The handler enqueues a signal and acks; it never commits inline.
- **`runInTenant` everywhere.** The scan command, the onboarding launch, the `RepoHealthReport` write, and the approve-commit Conductor action all run under `runInTenant` row-isolation ([`04b ¬ß11c`], [`07 ¬ßA`] lease) ‚Äî onboarding cannot reach another workspace's repo.
- **PTY-billing (interactive node-pty only).** The draft is a standard interactive `claude` PTY worker ([`07 ¬ßA`] step 7) ‚Äî `type:http` hooks + the structured channel bill the Max subscription exactly as the P0.5 spike proved for every worker. **Never** `claude -p`/Agent-SDK ([`01 ¬ß1`]).
- **LuckyStack conventions.** File-based `_api` for the control-API ops; `apiRequest` typed calls (no `as any`); injected `functions.*` + `functions.tryCatch.tryCatch`; reuse before authoring (the changes/editor surface, `FileDiffViewer`, the `AgentRole`/`OrchestratorCommandRegistry`/`InfoDoc` machinery are all reused, not rebuilt); i18n on every onboarding string; Tailwind tokens for the Health panel.
- **V1_SCOPE wins.** Onboarding is **V1 (light)** ‚Äî it extends the GENERATE leg the build phase already ships, adds no deferred surface (no built-in MR, no multi-repo, no auto-regen-on-push beyond the existing opt-in `ai:refresh-docs` trigger). It reuses the single-forge GitLab push and the single-provider Claude PTY. Nothing here builds a deferred capability.
